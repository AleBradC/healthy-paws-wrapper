## 1. SES DNS records (required for mail delivery)

Gmail and Outlook silently drop mail from any domain that doesn't authenticate
itself. Before sending a single password-reset or verification email through
SES, publish these three record sets at your DNS provider (Cloudflare, in our
case). All three are **required** in 2026 — DKIM alone is not enough.

### SPF (TXT record on the apex)

Only one SPF record per domain is permitted. If you already have one, merge
the `include:` clauses instead of adding a second TXT record.

| Name | Type | Value | TTL |
|------|------|-------|-----|
| `@`  | TXT  | `v=spf1 include:amazonses.com -all` | Auto |

`-all` is the strict fail policy. Use `~all` (softfail) only if you have other
unverified senders during a migration window.

### DKIM (3 CNAME records, generated by SES)

In the SES console, go to **Verified identities → your domain → Authentication →
Easy DKIM** and click "Set up". SES will display three CNAMEs of the form
`<token>._domainkey.yourdomain.com` pointing at `<token>.dkim.amazonses.com`.
Publish all three in Cloudflare with proxy **OFF** (DNS only — Cloudflare's
proxy doesn't pass CNAMEs intact).

Verification can take up to 72 hours but usually completes within a few
minutes.

### DMARC (TXT record on `_dmarc`)

| Name      | Type | Value | TTL |
|-----------|------|-------|-----|
| `_dmarc`  | TXT  | `v=DMARC1; p=quarantine; rua=mailto:dmarc@yourdomain.com; ruf=mailto:dmarc@yourdomain.com; pct=100; aspf=s; adkim=s` | Auto |

Start with `p=quarantine` for the first two weeks while you watch the
aggregate reports for misconfigured senders. Once everything is clean,
move to `p=reject`. Use `p=none` only during initial setup if you're not
yet receiving reports — never as a permanent state.

### MAIL FROM domain (optional but recommended)

In **SES → your identity → MAIL FROM domain** set `mail.yourdomain.com`.
SES will give you one extra MX and one extra TXT record to publish. This
aligns the SMTP envelope-from with your domain so SPF doesn't fall back
to `amazonses.com`.

### Move out of sandbox

Brand-new SES accounts can only send to verified addresses and are capped at
200 emails/day. Open a support case from the SES console requesting
**production access** with:
- Your domain
- Expected volume (be honest — they reject vague answers)
- Bounce/complaint handling plan (we have `MAIL_FROM` + SNS configured)
- Mailing-list opt-in description

Approval usually arrives within 24 hours.

## 2. Sentry

Both repos read their DSN from env:

- **Backend:** `SENTRY_DSN` in `healthy-paws-service/.env`
- **Frontend:** `VITE_SENTRY_DSN` baked into the build (it's public)

When the DSN is unset, Sentry init is a no-op — local dev and CI never
ship events. Create separate projects per environment (dev / staging /
prod) so error volumes don't cross-contaminate.

The free Developer plan gives 5k errors and 10k performance events per
month, which is comfortably above what a beta-traffic deployment will
generate.

## 3. Database backups (cron entry)

`scripts/backup.sh` runs on the **EC2 host** (not in a container) and
shells into the `db` service. Cron entry:

```bash
sudo crontab -e
# Daily at 03:00 UTC. Keep it off-peak; the dump locks tables briefly.
0 3 * * * AWS_REGION=eu-central-1 S3_BUCKET=healthy-paws-backups COMPOSE_DIR=/home/ec2-user/healthy-paws-wrapper DB_USER=healthy_paws DB_DATABASE=healthy_paws /home/ec2-user/healthy-paws-wrapper/scripts/backup.sh >> /var/log/healthy-paws-backup.log 2>&1
```

The bucket needs an S3 lifecycle policy: **transition to Glacier IR after
30 days, expire after 365 days**. The EC2 instance role needs `s3:PutObject`
on the bucket+prefix — no static AWS keys on the box.

Test the restore path at least once before you rely on the backups:

```bash
gunzip -c healthy-paws_2026-05-17T03-00-00Z.sql.gz \
  | docker compose exec -T db psql -U healthy_paws -d healthy_paws_restore_test
```

## 4. Database migrations

`docker compose up` automatically runs the one-shot `migrate` service before
`backend` boots. To run migrations manually:

```bash
# Apply pending migrations (idempotent — skips already-applied ones)
docker compose run --rm migrate

# Revert the most recent migration
docker compose run --rm migrate npm run migrate:down

# Create a new migration (run on the host, not in compose)
cd healthy-paws-service && npm run migrate:create -- describe-the-change
```
